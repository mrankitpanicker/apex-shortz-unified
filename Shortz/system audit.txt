5. System Architecture
Architecture Pattern: Monolithic Linear Pipeline with GUI Shell
The system follows a strictly sequential pipeline pattern. There is no parallelism, no task queue, no service mesh. Each stage runs to completion before the next begins.

PIPELINE TYPE:    Sequential, single-pass, synchronous within worker thread
PARALLELISM:      None (one video at a time)
COMMUNICATION:    In-process function calls (no IPC, no HTTP)
STATE MANAGEMENT: File-based (progress.txt, history.json)
GPU USAGE:        Sequential exclusive access (XTTS load → inference → unload implicit → Whisper load → inference)
GPU / CPU Resource Usage
Stage	Resource	Notes
XTTS Model Loading	GPU VRAM (~2–4 GB)	Loaded at module import time, persists in memory
TTS Synthesis	GPU (CUDA)	Per-chunk inference, ~250 chars at a time
Whisper Loading	GPU VRAM (~1 GB)	Loaded fresh every run inside 
main_generate()
Whisper Transcription	GPU (CUDA)	Single-pass transcription of full audio
FFmpeg Operations	CPU only	Subprocess calls, I/O bound
ASS Generation	CPU only	String formatting, negligible
PyQt6 GUI	CPU + GPU (compositor)	60 FPS wave animation
IMPORTANT

XTTS and Whisper coexist in GPU VRAM simultaneously. XTTS is loaded at module import and never unloaded. Whisper is loaded inside 
main_generate()
 for each run. On GPUs with ≤6 GB VRAM, this may cause OOM failures.

8. Performance Considerations
GPU Memory Management
CAUTION

Critical Issue: XTTS model is loaded at module import time (
Shortz.py
 lines 114–129), which occurs inside the worker thread via import Shortz. The model persists in GPU VRAM for the lifetime of the process. Whisper is loaded fresh per run (line 528) and also persists because there's no explicit del model or torch.cuda.empty_cache() call. This means:

After first run: XTTS (~2–4 GB) + Whisper (~1 GB) + PyTorch overhead = ~5+ GB VRAM
On a 6 GB GPU, this will OOM on the second run or during large audio transcription
Model Loading/Unloading
Model	When Loaded	When Unloaded	Problem
XTTS v2	Module import (once)	Never (process lifetime)	Acceptable for single-session use
Whisper Small	Every 
main_generate()
 call	Never (no explicit cleanup)	Accumulates VRAM across runs
Long-Running Process Concerns
Simulated progress bars in 
tts_generate_and_measure()
 (lines 254–260): A loop runs time.sleep(0.01) × 100 iterations = 1 second of artificial delay per chunk purely for visual progress bar output. For 3 chunks, this wastes 3 seconds.
5-second time.sleep() in 
complete_automation()
 (line 311): Blocks the controller thread for 5 seconds after writing the done file. Since this runs on the main thread via signal, it freezes the GUI for 5 seconds.
